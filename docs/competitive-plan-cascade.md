# Competitive Analysis: Scout-and-Wave vs. Plan Cascade

**Date:** 2026-03-31
**SAW version:** 0.76.0 (protocol), 0.92.x (Go SDK)
**Plan Cascade version:** 4.4.0

---

## 1. What plan-cascade is

Plan Cascade is an AI-powered development framework that decomposes tasks into parallel-executable stories using a three-tier planning model: Mega Plan (project-level, produces `mega-plan.json`), Hybrid Ralph (feature-level, produces `prd.json`), and Story Execution (parallel agent implementation). The core model is PRD-driven: a planning phase generates structured requirements documents, which drive batched parallel story execution with quality gates at each step. The framework is primarily a Claude Code plugin but also ships an MCP server for Cursor/Windsurf integration, a Python standalone CLI (in development), and a Tauri desktop app.

Plan Cascade's defining architectural choice is an auto-strategy selector (`/plan-cascade:auto`) that classifies tasks across four strategies — DIRECT, HYBRID_AUTO, HYBRID_WORKTREE, and MEGA_PLAN — using LLM-based structured analysis. Three execution flow depths (quick/standard/full) control gate strictness; full flow, which is the default, enforces hard DoR/DoD gates, mandatory TDD compliance, AI code review, and batch confirmations. Agents are heterogeneous: stories can be routed to claude-code (via Task tool), codex, aider, amp-code, or cursor-cli based on story type, phase, or manual override, with a configurable fallback chain.

Plan Cascade's target audience is teams who want a lower-friction entry point to AI-assisted parallel development, particularly for web frameworks (React, Vue, Rust) where it ships auto-detected framework skill injection. It is notably more accessible than SAW at the start — one command (`/plan-cascade:auto "..."`) covers the full surface from simple fixes to multi-feature projects — and places heavier emphasis on quality methodology (TDD, DoR/DoD, spec interviews) than on parallelization correctness guarantees.

## 2. Where SAW is stronger

**Structural conflict prevention vs. best-effort parallelism.** SAW's I1 invariant — disjoint file ownership, enforced before agents launch (E3), during execution (PreToolUse hooks), at completion (E42 SubagentStop), and at merge (E11 hunk-level prediction) — makes merge conflicts structurally impossible within a wave. Plan Cascade does not have file ownership assignment; parallel stories in the same batch can touch the same files. The system does not verify disjointness before launch. When conflicts occur, they surface at merge time with no preemptive structural guarantee.

**Protocol enforcement is mechanical, not advisory.** SAW runs 17 lifecycle hooks that enforce invariants at the tool boundary — a wave agent cannot write to a file it does not own because the `check_wave_ownership` PreToolUse hook rejects the call before it executes. Plan Cascade's execution model relies on model cooperation and progress file conventions (`progress.txt`, `.agent-status.json`). There is no hook layer blocking agents from making off-plan file changes during story execution.

**Interface contracts before parallelization (I2).** SAW's Scaffold Agent materializes shared types as committed source code before any wave agent launches. Agents compile against committed interfaces, not hallucinated signatures. Plan Cascade's design document (`design_doc.json`) provides architectural context injected into agent prompts, but it is documentation, not committed code. There is no scaffold phase that compiles and verifies shared types before parallel agents run.

**Wave sequencing with verified boundaries (I3).** SAW enforces that Wave N+1 does not launch until Wave N is merged and post-merge verification passes. This surfaces integration failures at wave boundaries rather than at the end. The engine checks this in `PrepareWave`: if any prior-wave agent lacks a `status: complete` completion report, the next wave is blocked. Plan Cascade's sequential batch model executes batch N before batch N+1, but there is no documented equivalent of post-merge build verification and integration gap detection (E25/E26) gating the next batch's launch.

**PROGRAM layer for multi-feature coordination.** SAW's PROGRAM layer (P1–P5 invariants) runs multiple features as a coordinated tier structure with cross-IMPL file ownership conflict detection (greedy graph coloring), IMPL branch isolation so partial work from one feature cannot contaminate another, and tier gate verification before tier N+1 launches. Plan Cascade's mega-plan batches features in worktrees with sequential batch execution, but does not perform cross-feature ownership conflict analysis before batches launch (no equivalent of `check-impl-conflicts`). Two features in the same mega-plan batch can own overlapping files.

**Go SDK as importable infrastructure.** SAW ships 30+ importable Go packages (`pkg/engine`, `pkg/protocol`, `pkg/solver`, `pkg/builddiag`, `pkg/codereview`, etc.) that can be embedded in any Go program. The CLI and web app are thin wrappers over the same SDK. Plan Cascade's Python core (`src/plan_cascade/`) is similarly structured, but the Python CLI is marked "in development" and unstable. SAW's Go SDK ships pre-built binaries for macOS/Linux/Windows (amd64/arm64) via GoReleaser and is already used in production.

**Wiring obligation tracking (E35).** SAW requires agents to declare `wiring:` blocks for every exported symbol needing downstream integration, then validates post-merge that all wiring obligations were fulfilled. `populate-integration-checklist` auto-generates the checklist from file ownership patterns. There is no equivalent in plan-cascade; integration correctness is left to agent judgment.

**Resume, retry, and failure context.** SAW's `resume-detect` identifies interrupted sessions with progress percentage, `build-retry-context` produces classified error context rather than raw stderr, and retried agents receive prior-work context from tool journals (E23A). Plan Cascade has resume commands (`/plan-cascade:resume`, `/plan-cascade:hybrid-resume`, `/plan-cascade:mega-resume`) that read state files to skip completed work, but the retry context injection is less structured — it surfaces error output from quality gates rather than classified, actionable fix context.

**Autonomy levels and daemon mode.** SAW supports graduated autonomy: supervised, semi-autonomous (--auto), and fully autonomous daemon mode with a queue and per-item `autonomy_override` and `require_review` flags. `sawtools daemon` runs continuous unattended execution from a queue with auto-remediation. Plan Cascade has `--auto-run` for unattended execution and a Python `auto-execute.py` script, but no persistent daemon/queue architecture.

**Observability with structured cost and performance events.** SAW's E40 event schema emits `cost` (token usage + USD estimates per agent), `agent_performance`, and `activity` events to a SQLite store with a query interface. The web app streams these over SSE. Plan Cascade's desktop app has an analytics dashboard with token/cost tracking, but the plugin-mode observability is limited to log files (`progress.txt`, `.agent-outputs/`) and the `/plan-cascade:dashboard` command which aggregates from JSON state files.

## 3. Where plan-cascade is stronger

**Auto-strategy classification lowers the entry cost.** `/plan-cascade:auto "description"` handles the full range from simple bug fixes to multi-feature projects in one command. The LLM-based strategy classifier produces a structured JSON analysis with confidence score, reasoning, and confirmation points, and routes to the correct workflow. SAW requires the user to understand and choose between `/saw scout` (single feature), `/saw program plan` (multi-feature), or `/saw bootstrap` (new project). The cognitive load at the entry point is higher in SAW.

**TDD as a first-class gate.** Plan Cascade has a `TDDComplianceGate` that verifies test files were modified alongside code changes, with per-story risk-based auto-enablement. The `--tdd on` default in FULL flow enforces red-green-refactor discipline at the story level. SAW has no equivalent gate. E20 scans for stub patterns after agent completion, but there is no gate that requires tests to be written or modified as part of wave execution.

**DoR/DoD gates are well-articulated.** Plan Cascade's Definition of Ready (verifies stories are well-formed before execution) and Definition of Done (verifies completion quality post-execution) are distinct, configurable gates with SOFT/HARD modes. The FULL flow enforces hard DoR gates that block execution if acceptance criteria are not testable. SAW's E37 critic gate reviews briefs for symbol accuracy and import conflicts before launch, but does not assess story-level readiness criteria (testable acceptance criteria, risk documentation, verification hints).

**Heterogeneous agent routing.** Plan Cascade routes different story types to different agent binaries: bugfix stories go to codex, refactor stories go to aider, feature stories go to claude-code. This leverages specialization across existing tools. The `agents.json` config file and `--impl-agent`/`--retry-agent` flags give granular per-phase control. SAW uses a single agent type per role (Scout, Wave Agent, etc.) all running the same model backend. There is no equivalent of routing story types to different CLI tools.

**Spec interview as planning-time disambiguation.** Plan Cascade's spec interview (`/plan-cascade:spec-plan`) is a structured, resumable pre-PRD interview that produces `spec.json` as the source of truth, then compiles it into `prd.json`. The interview supports `--first-principles` mode for probing foundational assumptions and `--max-questions` for length control. It runs in the orchestrator before any agent launches. SAW's E39 interview mode (`/saw interview`) produces a REQUIREMENTS.md for Scout consumption, which is similar in purpose but less structured — `spec.json` has a defined schema with `metadata`, `overview`, `architecture`, `interfaces`, and `decisions` sections, whereas REQUIREMENTS.md is freeform markdown.

**Framework-specific skill injection.** Plan Cascade ships a three-tier skill system (builtin/external/user, priority 1–200) with auto-detected framework injection: React/Next.js projects automatically receive `react-best-practices` and `web-design-guidelines`; Rust projects receive `rust-coding-guidelines`, `rust-ownership`, `rust-error-handling`, and `rust-concurrency`. These are pulled from community-maintained Git submodules (vercel-labs/agent-skills, vuejs-ai/skills, actionbook/rust-skills). Users can add custom skills from local paths or remote URLs with per-phase injection rules. SAW has no equivalent skill injection layer.

**Desktop app with independent capabilities.** Plan Cascade's Tauri desktop app (React + Rust) is a standalone coding environment with its own LLM backends (7+ providers), knowledge base (RAG with HNSW vector indexing), codebase indexing (Tree-sitter across 6 languages), timeline/checkpoint session version control with branching and rollback, visual agent pipeline composer (DAG canvas editor), and full git GUI (46 commands with AI commit messages). These capabilities exist independently of the Claude Code plugin. SAW's web app is real-time monitoring and orchestration but does not include a standalone coding environment, knowledge base, or visual pipeline composer.

**MCP server for non-Claude Code environments.** Plan Cascade ships a FastMCP server that exposes its operations to Cursor, Windsurf, and other MCP-compatible tools, with preconfigured setup in `mcp-configs/`. SAW is explicitly designed for Claude Code (with the Agent Skills standard as a portability path), but does not ship an MCP server that makes its operations available to MCP-compatible tools directly.

**Design document hierarchy with ADRs.** Plan Cascade auto-generates `design_doc.json` at project and feature levels, with ADR tracking (ADR-001 for project, ADR-F001 for features), component/responsibility/dependency mapping, and story-to-component mappings. Feature design docs inherit from project design docs. The design doc is injected into each story's agent context filtered to relevant components. SAW's IMPL doc captures interface contracts, file ownership, and dependency structure but does not generate or maintain a structured architectural decision record.

**Configurable quality gate granularity.** Plan Cascade's quality gate configuration in `prd.json` is per-gate configurable: each gate (`format`, `typecheck`, `test`, `lint`, `tdd_compliance`, `code_review`, `implementation_verify`) has independent `required`, `check_only`, `min_score`, and `block_on_critical` fields. `fail_fast` controls whether the gate sequence aborts on first failure. SAW's quality gates are configured in `saw.config.json` but the per-gate configurability is less granular — gates run as part of `finalize-wave` and the gate cache (E38) handles idempotency, but the per-gate config surface exposed to users is narrower.

## 4. What SAW could absorb

**TDD compliance gate.** Add a quality gate that verifies test files were modified alongside implementation files in a wave. This is E20 extended from stub detection to test co-modification detection. It would run as part of `finalize-wave` and could be configured per-project in `saw.config.json`. The highest-value variant: check that for each agent's owned source files, a corresponding test file was also modified. This doesn't require adopting plan-cascade's full red-green-refactor prompting — just the completion verification.

**Strategy explainer as a first-class command.** `/saw scout --explain "feature"` could run the Scout's suitability analysis and output the structured analysis (suitability verdict, dependency count, estimated wave structure, estimated parallelization value) without committing to execution. SAW already has `analyze-suitability` and `sawtools run-scout` with suitability gating, but there is no lightweight "show me what Scout would plan without running it" entry point. Plan Cascade's `--explain --json` flag pattern is well-designed for this.

**DoR checks against IMPL briefs before agent launch.** Plan Cascade's Definition of Ready gate validates that stories have testable acceptance criteria, explicit risks, and verification hints. SAW's E37 critic gate checks symbol accuracy and ownership gaps — that is the *technical* pre-flight. A complementary gate checking whether each agent brief has a measurable verification command, declared risks, and non-ambiguous acceptance criteria would catch a different class of planning errors. This could be added as an E37 check type or a new E46 gate.

**Structured spec.json as Scout input.** Plan Cascade's `spec.json` schema (metadata, overview, architecture, interfaces, decisions, story_mappings) is more structured than SAW's REQUIREMENTS.md freeform markdown. Converting E39 interview output to a structured format with an explicit `verification_commands` field and `risk_level` per story would give the Scout better-typed input and enable programmatic validation. The `SpecQualityGate` concept (validating that criteria are testable before planning proceeds) is worth borrowing.

**Phase-based agent routing.** SAW currently uses per-role model configuration (Scout/Wave/Critic/etc.) but all agents within a role use the same model. Adding story-type-based routing — sending refactor-heavy wave briefs to an aider backend, or bug-fix briefs to a codex backend — would make SAW's agent selection more flexible. The implementation would be a new field in the wave agent's brief (`preferred_agent_type`) and a routing table in `saw.config.json`. Highest value for teams already using multiple CLI tools.

**`/saw auto "description"` as a lower-friction entry point.** A single command that classifies the request and routes to `scout` (single feature), `program plan` (multi-feature), or direct execution (too small to parallelize) would reduce the cognitive load at the start of a session. The classification logic could reuse SAW's existing `analyze-suitability` output and a simple feature-count heuristic. This does not require the LLM-based confidence-scored analysis plan-cascade uses — SAW's quantitative suitability scoring already provides the signal needed.

**Skill injection for framework-specific wave context.** Injecting framework-specific best-practice content into wave agent briefs based on detected project type (React, Rust, Go, etc.) would improve generated code quality without requiring Scout to rediscover the same conventions. SAW's conditional injection architecture (inject-agent-context script) already supports adding content to agent prompts based on conditions — extending it to framework detection is a natural fit. The actual skill content could pull from the same community submodules plan-cascade already uses.

**ADR tracking in IMPL docs.** Plan Cascade's design document records architectural decisions with context, rationale, and alternatives considered. Adding an optional `decisions:` section to the IMPL doc format — populated by Scout when it makes significant architectural choices — would give the IMPL doc more durable value as a project artifact. Currently the IMPL doc captures *what* was built (file ownership, interfaces, wave structure) but not *why* design choices were made. The `decisions:` section would close that gap for teams that use IMPL docs as post-hoc design records.

## 5. What plan-cascade could learn from SAW

**Structural conflict prevention is worth the overhead.** Plan Cascade's parallel story execution has no mechanism preventing two stories in the same batch from touching the same file. At small batch sizes this is rarely a problem; at 8-15 parallel stories it becomes probabilistic. SAW's I1 invariant — assigning every file to exactly one agent before launch — is the mechanism that makes the parallelism actually safe. The overhead is real (Scout phase, ownership table construction) but the correctness guarantee is qualitatively different from optimistic parallelism.

**Tool-boundary enforcement is more reliable than agent cooperation.** Plan Cascade relies on agents following their story prompts. SAW's PreToolUse hooks reject writes to unowned files before the tool executes. The difference is that hook-based enforcement does not degrade with model temperature, prompt injection, or context window pressure. For frameworks aiming at production reliability rather than prototyping convenience, this architecture gap matters.

**Wave boundary verification gates prevent cascade failures.** Plan Cascade advances to the next batch when quality gates pass on the current batch's stories. SAW also runs post-merge build verification to catch cross-agent integration failures that individual agents cannot see from their isolated worktrees — specifically, "Agent A changed a function signature and Agent B's code breaks at merge time even though both passed isolated tests." Plan Cascade's current gate model (verify per story, then merge) does not include a post-merge cross-agent build check.

**The IMPL doc as a git-tracked coordination artifact has lasting value.** Plan Cascade's `prd.json` and `mega-plan.json` are planning files added to `.gitignore` by default — they are ephemeral. SAW's IMPL doc is committed to the repository, machine-parseable, and contains the full execution history including completion reports, critic verdicts, gate results, and context injection metadata. This makes the IMPL doc reviewable, diffable, and useful as a post-mortem artifact. Plan Cascade's design document (`design_doc.json`) is committed, but execution state is not.

**Quantitative suitability scoring catches bad decompositions before they run.** SAW's five-precondition suitability gate (decomposability, interface discoverability, no investigation-first blockers, pre-scan of existing work, positive parallelization value) produces a dimensional score with a named failing precondition and a suggested alternative. A NOT_SUITABLE verdict with "P2: investigation-first blocker detected — suggest: explore before decomposing" is more actionable than a high-confidence DIRECT selection that still runs the full story pipeline on a task that cannot meaningfully be parallelized.

**The daemon/queue architecture enables production integration.** Plan Cascade's `--auto-run` flag enables unattended execution but requires an active Claude Code session. SAW's daemon mode polls a queue file, executes items autonomously, and supports per-item `require_review` overrides. This makes SAW integrable into CI/CD pipelines and background automation without a user session, which is a different deployment model than anything plan-cascade currently supports.
