## Multi-Gen Agent ID Consistency

The protocol adopted the `[A-Z][2-9]?` agent ID format (documented in `protocol/message-formats.md`, `protocol/execution-rules.md` E5/E16B, and both scout prompt files). The following files have inconsistencies:

### Issue 1 — `docs/IMPL/IMPL-e16-presence-grammar.md`: Old single-char `[A-Z]` regex throughout

Multiple lines in this historical IMPL doc reference the validator grammar using the **old** single-character-only `[A-Z]` pattern (without `[2-9]?`). Since this IMPL doc was the source-of-truth for the E16B grammar, and the current `protocol/execution-rules.md` E16B now documents `[A-Z][2-9]?`, the IMPL doc is stale.

Affected lines (representative, not exhaustive):
- Line 138: `At least one agent entry matching \[[A-Z]\]`
- Line 139: `Each agent entry (line matching ^\s+\[([A-Z])\])`
- Line 146: `At least one line matching [A-Z] agent pattern (bracket-enclosed uppercase letter): \[[A-Z]\]`
- Line 263: `detect plain fenced blocks ... that contain both a [A-Z] pattern and the word Wave`
- Line 320: `if echo "$e16c_block_content" | grep -qE "\[[A-Z]\]"`
- Line 434, 605, 794, 827: additional `[A-Z]` references

**Assessment:** This is a historical IMPL doc — it records what was designed and delivered, not the active protocol. No change required in the IMPL doc. The *active* protocol doc (`protocol/execution-rules.md` E16B, E16C) already uses `[A-Z][2-9]?` correctly. **No fix needed; document is archival.**

### Issue 2 — `implementations/claude-code/prompts/scaffold-agent.md`: E-rule range references only E1–E16

Line 12 of `scaffold-agent.md` reads:
```
(E1–E16) defined in `protocol/execution-rules.md`. Each is embedded verbatim at its point of
```

The current protocol defines E1–E19. The scaffold agent does not reference E17 (scout reads project memory), E18 (orchestrator updates project memory), or E19 (failure type decision tree). None of these rules directly govern the Scaffold Agent's behavior, but the cross-reference range is stale — it should read `E1–E19` to match the range stated in `agent-template.md` (which has already been updated).

**Severity:** Low. The Scaffold Agent's behavior is not affected by E17/E18/E19.

### Issue 3 — `implementations/claude-code/prompts/saw-skill.md`: E-rule range references only E1–E16

Line 39 of `saw-skill.md` reads:
```
(E1–E16) defined in `protocol/invariants.md` and `protocol/execution-rules.md`.
```

This is the Orchestrator skill, which is directly responsible for executing E17, E18, and E19 (project memory read/write, failure type decision tree). The range should read `E1–E19`.

Note: The body of `saw-skill.md` *does* reference E17, E18, E19 correctly in the execution logic (steps 6 and 4 of the wave loop). Only the preamble cross-reference anchor is stale.

**Severity:** Medium. The preamble is the first thing the Orchestrator reads to understand the rule space.
